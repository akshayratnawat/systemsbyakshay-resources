Concept 01 of 08

Vectors & Embeddings

An embedding is a coordinate in meaning space, not a magic blob of numbers.
The intuition

A vector is just an ordered list of numbers, but in AI it helps to picture it as a location in a space. If one axis roughly captures sentiment, another captures topic, another captures formality, then a sentence or song can sit somewhere inside that giant space. Nearby points mean similar meaning. Far points mean different meaning.

That is why embeddings feel so useful. They turn messy things like text, users, products, and songs into coordinates you can compare. OpenAI's text-embedding-3-large uses a 3072-dimensional vector, which sounds abstract until you remember that each dimension only needs to help separate meaning a little bit. The full geometry is what matters.

Once you see embeddings as coordinates, a lot of product behavior stops feeling mysterious. Search becomes nearest-neighbor lookup. Recommendations become finding users and items that occupy similar neighborhoods. Clustering becomes asking which regions of the space are naturally dense.

Close points share meaning. The axes are not human-labeled, but the geometry is still real enough to drive retrieval and ranking.
In production

OpenAI embedding APIs turn documents into dense vectors so RAG systems can retrieve semantically related chunks instead of keyword matches.

Spotify-style recommendation systems use Word2Vec-like embeddings to place users, songs, and artists in a shared space, then rank what sits nearby.

The math
cos(a, b) = (a · b) / (||a|| ||b||)
Cosine similarity asks whether two embedding directions line up, independent of raw magnitude.
  • a, b - two embedding vectors you want to compare.
  • a · b - the dot product, which grows when matching dimensions point in the same direction.
  • ||a||, ||b|| - vector magnitudes used to normalize away size.
  • cos(a, b) - a score near 1 means highly aligned, near 0 means unrelated, near -1 means opposite direction.
PyTorch example
import torch
import torch.nn.functional as F

embedding = torch.nn.Embedding(num_embeddings=10000, embedding_dim=3072)
ids = torch.tensor([7, 42])
vecs = embedding(ids)
score = F.cosine_similarity(vecs[0].unsqueeze(0), vecs[1].unsqueeze(0))
print(score.item())
Common mistake
Using raw dot product for retrieval when vectors have different magnitudes
This silently biases results toward embeddings with bigger norms, not better semantic match. In production that shows up as weird retrieval, popular-item bias, or chunks that look similar only because their vector length is large.
Normalize embeddings or use cosine similarity when your ranking goal is semantic closeness rather than energy or confidence.
Concept 02 of 08

Matrices & Tensor Shapes

A matrix is a container. A tensor is the same idea with more axes.
The intuition

The biggest beginner mistake in deep learning is not hard math. It is forgetting what each axis means. A tensor shape is really a set of labeled slots: batch, sequence, heads, channels, embedding dimension, height, width. If you lose track of the labels, your model can still run while learning garbage.

Think of a matrix as a spreadsheet with structure. Rows and columns are not interchangeable unless the operation says they are. A tensor is just the same idea extended to more directions. A Transformer batch often looks like (batch, seq, dim). An image batch often looks like (batch, channel, height, width). Same concept, different labels.

Production ML engineering is full of shape bookkeeping. You reshape for batching, permute for attention, flatten for classifiers, and unsqueeze for broadcasting. The people who debug fastest are usually the people who keep the axis meaning in their head at every line.

Diagram - labeled axes
batch = 32
seq = 128
dim = 768
image batch
channel = 3
height × width
The numbers matter, but the labels matter more. Most silent bugs come from preserving numbers while scrambling semantics.
In production

PyTorch training code and Hugging Face loops constantly move between batch-first and head-first layouts. Attention layers commonly reshape from (B, S, D) to (B, H, S, d).

Many nasty bugs are not exceptions. They are tensors with valid sizes but wrong axis meaning, so the training loop keeps going and quality quietly collapses.

The math
(m, k) × (k, n) → (m, n)
Matrix operations are legal only when the shared dimensions line up. The output keeps the outer dimensions.
  • m - number of rows in the first matrix.
  • k - inner dimension that must match across both operands.
  • n - number of columns in the second matrix.
  • (m, n) - output shape after combining rows from the first with columns from the second.
PyTorch example
import torch

x = torch.randn(4, 128, 256)  # batch, seq, dim
print(x.shape)
y = x.view(4, 128, 8, 32).permute(0, 2, 1, 3)
print(y.shape)  # batch, heads, seq, head_dim
Common mistake
Swapping sequence and feature dimensions in attention code
The tensor still has the same total number of elements, so the model does not always crash. Instead, attention operates across the wrong axis and training is quietly poisoned. The result looks like unexplained instability or a model that never really learns.
Write shapes in comments, assert tensor sizes at critical boundaries, and treat (batch, seq, dim) as semantic labels instead of just integers.
Concept 03 of 08

Matrix Multiplication

This is the core operation behind almost every neural network layer.
The intuition

If you only memorize one equation in deep learning, make it y = Wx + b. That is a linear layer. The matrix W says how much each input feature should influence each output feature. The vector b shifts the result. Stack enough of these with nonlinearities and you get most modern networks.

Matrix multiplication looks intimidating because it is written compactly, but the behavior is simple: each output neuron is a weighted sum of inputs. One row of W defines one detector. Multiply that row by the input, add everything up, and you get one output value.

This is why GPUs matter so much. Training and inference are mostly giant matrix multiplies repeated at ridiculous scale. NVIDIA tensor cores exist because AI workloads are basically a relentless stream of linear algebra. On an A100, those matmuls are what turn model architecture into actual throughput.

Every row of W is a recipe for one output feature. Neural networks feel fancy, but this is the engine.
In production

NVIDIA A100 tensor cores are built to accelerate matrix multiplies at enormous scale. Roughly speaking, modern AI hardware is optimized around this one primitive.

When a model is slow, you are often really asking how efficiently your stack feeds these matmuls to the GPU.

The math
y[i] = Σj W[i, j] x[j] + b[i]
The i-th output is the weighted sum of every input feature plus its own bias term.
  • W[i, j] - weight connecting input feature j to output feature i.
  • x[j] - value of input feature j.
  • Σ - add contributions from all input features.
  • b[i] - bias for output feature i.
  • y[i] - final activation before any nonlinearity.
PyTorch example
import torch
import torch.nn as nn

x = torch.randn(32, 768)
layer = nn.Linear(768, 256)
y1 = layer(x)
y2 = torch.matmul(x, layer.weight.T) + layer.bias
print(torch.allclose(y1, y2))
Common mistake
Forgetting that inner dimensions must match
This is the classic shape mismatch error: you try to multiply (batch, 512) by a weight matrix expecting 768 features. In a more subtle version, you reshape tensors until the code runs but the learned mapping is wrong.
Track the feature dimension through the network and verify it at every layer boundary before blaming the optimizer or dataset.
Concept 04 of 08

Dot Product & Cosine Similarity

Dot product measures alignment. Attention turns those alignments into weights.
The intuition

The dot product answers a simple question: how much do these two vectors point in the same direction? If two vectors align, the score is high. If they are orthogonal, the score is near zero. If they point against each other, the score goes negative.

This matters because attention is built from dot products. A query vector asks what it is looking for. Key vectors say what each token contains. The dot product between them becomes a compatibility score. Softmax turns those scores into probabilities, and the model blends values accordingly.

Cosine similarity is just the normalized version of the same idea. Retrieval systems like Pinecone and Weaviate often use cosine when semantic direction matters more than vector magnitude. Transformers, meanwhile, often use raw dot products but scale them to keep the softmax stable.

Attention is similarity scoring plus normalization. The glamour is in the scale. The mechanic is still alignment.
In production

Transformers compute softmax(QKᵀ / √dk)V on every layer. That one line powers chatbots, copilots, and RAG rerankers.

Vector databases use the same geometric idea for retrieval, even when the serving layer hides the math behind an API.

The math
a · b = ||a|| ||b|| cos(θ)
Attention(Q, K, V) = softmax(QKᵀ / √dk)V
The first equation explains alignment. The second shows how a whole attention head uses that alignment to blend values.
  • θ - angle between vectors. Smaller angle means stronger alignment.
  • Q - query matrix containing what each token wants.
  • K - key matrix describing what each token offers.
  • V - value matrix containing the information to be blended.
  • dk - key dimension used to scale scores before softmax.
PyTorch example
import math
import torch

q = torch.randn(4)
k = torch.randn(4)
print(torch.dot(q, k).item())
Q, K, V = [torch.randn(2, 3, 8) for _ in range(3)]
weights = torch.softmax(torch.matmul(Q, K.transpose(-1, -2)) / math.sqrt(Q.size(-1)), dim=-1)
out = torch.matmul(weights, V)
Common mistake
Skipping the √dk scaling in attention
As vector dimension grows, raw dot products get large. Then softmax saturates, one token wins too hard, gradients get ugly, and training becomes brittle. The model looks overconfident for a mathematical reason, not a mysterious one.
Always scale by the square root of key dimension before softmax so score variance stays in a healthy range.
Concept 05 of 08

Norms - L1 and L2

Norms tell you how big a vector is, and that makes them a health metric for training.
The intuition

A norm is just a way to measure size. For vectors, that means asking how large the overall signal is once you combine all coordinates. In training, the most important vector is often the gradient. If the gradient norm suddenly spikes, you are usually watching the system become unstable before the loss graph fully reveals it.

L2 norm is the familiar Euclidean length. It grows faster when individual coordinates get very large, which makes it useful for spotting exploding gradients. L1 norm adds absolute values instead, which is more robust and often used when you want sparsity or a less outlier-sensitive notion of size.

This is why gradient clipping works. You are not magically fixing optimization. You are enforcing a maximum update size so one bad batch cannot kick the model into a broken region. GPT-style training stacks commonly clip gradient norm at values like 1.0 for exactly this reason.

Diagram - norm monitoring

Loss often tells you after the crash. Gradient norm warns you on the way there.
In production

Large-model training pipelines routinely log gradient norms alongside loss and learning rate. It is one of the earliest signals of instability.

Gradient clipping such as max_grad_norm = 1.0 became standard because it prevents rare but catastrophic parameter updates.

The math
||x||2 = √Σ xi2
||x||1 = Σ |xi|
L2 measures Euclidean length. L1 measures total absolute mass.
  • xi - the i-th coordinate of vector x.
  • Σ - sum over all coordinates.
  • ||x||2 - L2 norm, sensitive to large coordinates.
  • ||x||1 - L1 norm, often linked to sparsity and robust magnitude measurement.
PyTorch example
import torch
from torch.nn.utils import clip_grad_norm_

model = torch.nn.Linear(16, 4)
x = torch.randn(8, 16)
loss = model(x).pow(2).mean()
loss.backward()
print(torch.norm(model.weight.grad, p=2).item())
clip_grad_norm_(model.parameters(), max_norm=1.0)
Common mistake
Only watching loss while gradients are exploding underneath
Teams often notice NaNs only after the training run is already ruined. The actual root cause was visible much earlier in the gradient norm trace. Loss is a lagging indicator here.
Log gradient norms per step or per interval, clip when appropriate, and treat sudden spikes as a first-class debugging signal.

Concept 06 of 08

Linear Transformations

A weight matrix does not just compute numbers. It warps space.
The intuition

When you multiply by a matrix W, you are applying a transformation to the whole input space. Points can rotate, stretch, compress, or shear. In ML terms, the model is learning how to remap features so the next decision becomes easier.

This geometric view is more useful than thinking about isolated neurons. A classifier head learns a transformation that pushes classes apart. Early layers in a network reshape raw inputs into feature spaces where useful patterns line up more cleanly. Depth means applying many warps in sequence.

That is also why deep networks need architectural help. Without skip connections, repeated warping can squeeze signal into bad regions where gradients fade. Residual paths keep a stable route through the stack so the model can learn useful transforms without destroying information every time.

Training is really a search for transforms that make the task linearly easier at the next step.
In production

Classifier layers in vision and language models learn to move examples of different classes apart so a simple final decision boundary works.

Residual networks and Transformers use skip connections because deep stacks of linear transforms plus nonlinearities are hard to optimize without shortcuts.

The math
y = Wx
The same input vector space is being remapped into a new coordinate system defined by W.
  • x - point in the original feature space.
  • W - learned transformation matrix.
  • y - transformed point in the new space.
  • Geometric effect - depending on W, this can rotate, scale, reflect, or shear the whole space.
PyTorch example
import torch
import torch.nn as nn

layer = nn.Linear(2, 2, bias=False)
layer.weight.data = torch.tensor([[2.0, 0.5], [0.0, 1.5]])
points = torch.tensor([[1.0, 0.0], [0.0, 1.0], [1.0, 1.0]])
print(layer.weight)
print(layer(points))
Common mistake
Building deep stacks of transforms without skip connections
Each layer can distort the signal a little. After many layers, useful gradient paths disappear and optimization gets harder. The issue is not that linear algebra stopped working. It is that repeated warps without a clean fallback path make learning fragile.
Use residual connections in deep models so the network can learn transforms on top of a stable identity path instead of replacing it every time.
Concept 07 of 08

SVD & LoRA

Low-rank adaptation works because many useful weight updates live in a much smaller subspace than the full matrix.
The intuition

Singular Value Decomposition says any matrix can be broken into three pieces: rotate, scale, rotate. That matters because it gives you a clean way to describe where the real action of a matrix lives. Often, only a few singular directions carry most of the useful energy.

LoRA takes that idea and turns it into a deployment trick. Instead of updating a giant weight matrix directly, it learns two skinny matrices whose product is a low-rank update. If task-specific change mostly lives in a few directions, you get most of the benefit with a tiny fraction of the parameters.

That is why parameter-efficient fine-tuning changed the economics of model adaptation. A base model can stay frozen while a small low-rank adapter captures the task delta. QLoRA pushes this even further by quantizing the base model and training only the compact adapters.

SVD tells you how a matrix spends its capacity. LoRA bets that task-specific change only needs a few of those directions.
In production

LoRA can shrink trainable parameter count from millions to tens of thousands on a given layer stack, often cutting adaptation cost by two orders of magnitude.

QLoRA made it practical to fine-tune very large models, including 65B-scale systems, on hardware budgets that would have looked unrealistic before.

The math
A = UΣVᵀ
ΔW = BA, where rank(ΔW) << rank(W)
SVD decomposes a matrix into orthogonal directions and strengths. LoRA constrains the learned update to a low-rank product.
  • U - left singular vectors, or output-side directions.
  • Σ - diagonal matrix of singular values showing how important each direction is.
  • Vᵀ - right singular vectors, or input-side directions.
  • ΔW - trainable adapter update added to a frozen base weight matrix.
  • rank - number of independent directions available to represent change.
PyTorch example
import torch

W = torch.randn(8, 8)
U, S, Vh = torch.linalg.svd(W)
r = 2
A = torch.randn(r, W.size(1))
B = torch.randn(W.size(0), r)
delta_W = torch.matmul(B, A)
print(S[:r], delta_W.shape)  # LoRA-style low-rank update
Common mistake
Choosing LoRA rank so low that the adapter cannot express the task
A rank like r = 2 can be great for lightweight tasks and terrible for complex reasoning, domain transfer, or multilingual behavior. The training run looks cheap, but the adapter simply underfits because it does not have enough independent directions to move in.
Treat LoRA rank as a capacity knob. Start from the task complexity, monitor validation quality, and increase rank when the adapter is clearly too constrained.
Concept 08 of 08

Eigenvectors & PCA

Some directions survive a transformation. Those directions often explain the whole system.
The intuition

An eigenvector is a direction a matrix does not rotate away from. Apply the transformation and the vector stays on the same line - it only gets stretched or shrunk. That sounds niche, but it is the key to understanding long-run system behavior.

Google's original PageRank can be understood through the dominant eigenvector of a link transition matrix. Repeatedly apply the web graph dynamics and one importance pattern keeps reappearing. In representation work, PCA uses closely related spectral ideas to find the directions with the most variance so you can compress embeddings without throwing away as much structure.

The practical lesson is that not all directions in data are equally important. Some directions dominate signal. Others mostly carry noise or redundancy. Spectral methods help you find the axes the system naturally cares about.

Eigenvectors expose persistent directions. PCA uses top directions to keep information while shrinking dimension.
In production

PageRank relies on the dominant eigenvector of a graph-related matrix to estimate stable importance across the web.

PCA is still used for embedding compression, visualization, denoising, and feature reduction before faster downstream models.

The math
Av = λv
After transforming v by matrix A, the direction stays the same and only the scale changes by λ.
  • A - transformation matrix.
  • v - eigenvector, a direction preserved by the transformation.
  • λ - eigenvalue, telling you how strongly that direction is amplified or shrunk.
  • PCA link - principal components are eigenvectors of the covariance matrix, ordered by eigenvalue size.
PyTorch example
import torch
from sklearn.decomposition import PCA

A = torch.tensor([[2.0, 1.0], [1.0, 2.0]])
eigvals, eigvecs = torch.linalg.eig(A)
X = torch.randn(100, 16)
pca = PCA(n_components=4)
X_reduced = pca.fit_transform(X.numpy())
print(eigvals.real, X_reduced.shape)
Common mistake
Running PCA on unnormalized features
PCA follows variance. If one feature has a giant scale, the top eigenvectors will mostly track that scale rather than genuine information content. You think you compressed signal, but you mostly encoded units.
Standardize or otherwise normalize features before PCA unless raw scale itself is the thing you intentionally want to preserve.
