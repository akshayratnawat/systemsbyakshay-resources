Temperature Settings Reference Card

The complete guide to LLM sampling parameters - from the logit math to provider-specific defaults

1. The Core Mechanism

Temperature is a single division operation applied to raw model scores (logits) before they become probabilities.

The Formula
p(token) = softmax( logit / T )
TemperatureEffect on DistributionWhat Happens
T = 0Collapsed to single tokenGreedy decoding. Always picks the highest logit. Deterministic.
T = 0.1-0.3Very sharp peakNear-deterministic. Top token dominates. Minor variation possible.
T = 0.5-0.7Moderate spreadBalanced. Top 3-5 tokens share meaningful probability mass.
T = 1.0Model's native distributionUnmodified probabilities. This is the raw model behavior.
T > 1.0Flattened distributionAll tokens approach equal probability. High hallucination risk.
Key insight: T < 1 amplifies score differences (makes the model more confident). T > 1 compresses them (makes the model more uncertain). "Creative" is a human label for what uncertainty produces.

The #1 production rule: If users will act on the output (make decisions, follow instructions, trust the information), temperature should be as low as your task allows.

The trap: OpenAI and Anthropic both default to T=1.0. If you're using their SDKs without explicitly setting temperature, every call runs at maximum variance. This is the most common source of "inconsistent outputs" bugs.

4. The Sampling Triad - How Parameters Compose

Temperature, top-k, and top-p are applied in sequence. Understanding the order matters.

Processing Order

StepParameterWhat It Does
1TemperatureDivides all logits by T. Reshapes the entire distribution. Happens first.
2Top-kHard cutoff: keep only the k highest-probability tokens. Everything else gets zero probability.
3Top-pDynamic cutoff: keep tokens until cumulative probability reaches p. Adapts to context - sometimes 5 tokens, sometimes 50.
4SampleRandom sample from the remaining distribution.
Mental model: Temperature changes the shape. Top-k sets a hard floor. Top-p adapts dynamically. They're three different tools solving three different problems.

Recommended Combinations

Use CaseTtop-ptop-kWhy This Combo

Controlled variation0.30.920Narrow distribution + tight truncation. Safe for production.
Balanced chat0.70.950Standard production setting. Varied but not risky.
Creative / brainstorm0.90.95100Wide distribution, loose truncation. Maximum diversity.

5. Why Temperature Errors Compound

Temperature-induced hallucination is not a single-token problem. It's a cascade.

StepWhat Happens
1High T makes a low-scoring token plausible enough to sample.
2That wrong token shifts the context for the next prediction.
3In the new context, what WAS the "right" next token may no longer score highest.
4The model now generates fluent text in the wrong direction. It sounds confident and is wrong.
5Each subsequent token is conditioned on the accumulating errors. The hallucination looks intentional.
Debugging rule: If your model is hallucinating, drop temperature to 0.1 first. If hallucinations disappear, it was sampling variance, not a model knowledge problem. If they persist, the issue is upstream (retrieval, prompt, or training data).

6. Temperature Architecture Checklist

PracticeWhy
☐Set temperature explicitly on every API callNever rely on provider defaults (most are T=1.0)
☐Use per-route temperature in your LLM GatewayDifferent tasks need different settings. Centralize control.
☐Log temperature with every generationYou cannot diagnose quality regressions without this.
☐Use T=0 for all structured output (JSON, XML)Any variation breaks parsers. No exceptions.
☐Include temperature in prompt version registryWhen you change T, log it as a prompt version change.
☐A/B test temperature changes like any other parameterSmall T changes can have large quality impacts.
☐Set top-p and top-k alongside temperatureThey compose. Setting one without the others is incomplete.

7. Interview Quick Reference

QuestionStrong Answer
"What does temperature do mathematically?"Divides all logits by T before softmax. T<1 amplifies score gaps (more deterministic). T>1 compresses them (flatter, riskier). Formula: p = softmax(logit/T).
"Your support bot hallucinates order numbers at T=0.9. First move?"Drop T to 0.1, re-run same 50 queries, compare exact-match rate. If hallucinations drop, it's temperature-induced compounding. If not, problem is upstream.
"Why not T=0 for everything?"T=0 is greedy - always picks top logit. Works for extraction but produces repetitive text for chat. Use per-route temperature: T=0 for JSON, T=0.3 for code, T=0.7 for conversation.
"Difference between temperature and top-p?"Temperature reshapes the full distribution. Top-p truncates it to the minimum token set covering p% probability. They compose: temperature first, then top-p.
"How do temperature, top-k, and top-p interact?"Applied in order: (1) T divides logits, (2) top-k hard-cuts to k tokens, (3) top-p dynamically trims to cumulative p%. Three tools, three problems.
