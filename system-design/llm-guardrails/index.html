LLM Guardrails Architecture

Why "Don't Do Harmful Things" isn't enough - the 3-layer architecture that actually enforces safety

1. The Core Problem

"How do you prevent your AI system from being manipulated into producing harmful content?" It's a standard system design interview question. Most candidates answer with a system prompt: "I add safety instructions to tell the model what not to do."

Senior engineers describe an architecture. Not because the prompt doesn't matter - it does - but because they understand something fundamental about how LLMs process inputs. The model you're trying to protect against manipulation is not a reliable defender against it. The system prompt and the attacker's injected instruction sit in the same context window, processed in the same attention pass. One of them wins. You cannot know in advance which one.

The Fundamental Constraint
system_prompt ∈ context_window ← attacker_input ∈ context_window

Real guardrails are not instructions to the model. They're structural components that run before the model and after it, implemented in code that cannot be jailbroken.

7 of the OWASP LLM Top 10 (2025 v2.0) security risks are guardrail problems - not model problems. You can't prompt-engineer your way out of them. They require architectural defense.

2. Input Guard - Stop the Attack Before the Model Sees It

The Input Guard is your first line of defense, and the most important. It runs before the LLM ever sees the user's message. It does not involve the main model - it's fast, independent, deterministic code. Three checks, in order:

CheckMechanismLatencyWhat It Prevents
PII DetectionPattern + NER classifier~5msNames, emails, SSNs, credit cards entering the context. Scrub (consumer) or block (HIPAA).
Injection ClassifierML classifier (not regex)~20ms"Ignore previous instructions," novel prompt injection patterns. Generalizes beyond known attacks.
Topic ScopeDomain classifier~10msOut-of-scope requests. Rejected before model call - costs zero inference tokens.

Three checks run before the LLM sees anything. None involve the main model. All are code - they cannot be bypassed by clever prompting.

The production rule: Use a classifier for injection detection, not a regex. Rules catch known patterns in ~5ms. A classifier catches novel attacks in ~20ms. Novel attacks are the ones that matter.

3. Prompt Guard - Constrain Execution, Not Just Output

The Prompt Guard sits between the Input Guard and the LLM. Its job is not to filter bad content - that's the Input Guard's job. The Prompt Guard constrains what the LLM is structurally capable of doing during this request.

ConstraintWhere EnforcedWhat It Prevents
Tool PermissionsOrchestratorRead-only user gets an orchestrator that won't call write-capable tools. Model can't "get around" it - the permission isn't in the model's reach.
Context Window BudgetGatewayHard limit (e.g. 8K tokens) on user input. Prevents pushing system prompt to low-attention positions via extremely long inputs.
Output FormatSchema validatorRequiring structured JSON with a defined schema prevents format confusion attacks on downstream systems.

The Prompt Guard constrains what can happen, not just what's requested. Tool permissions live in the orchestrator - the model can't reach tools it hasn't been granted.

The production rule: If the model can't call the tool, it doesn't matter what it generates. Tool permission enforcement lives in the orchestrator, checked before execution - not in the system prompt, where it's just advice.

4. Output Guard - After the Model, Before the User

The Output Guard is your last line of defense. Even with a strong Input Guard and tight Prompt Guard, edge cases reach the model. Novel jailbreaks. Unusual inputs that slip past the injection classifier. The Output Guard exists to catch what the earlier layers missed - and to do so before the response ever reaches the user.

CheckMechanismLatencyWhat It Catches
Harm ClassifierExternal (Llama Guard / Azure Content Safety)~50msHarmful content the model was manipulated into producing. Never the main model self-assessing.
PII LeakagePattern + NER on output~5msModel hallucinating or leaking personal data - even when input PII was scrubbed at Layer 1.
Faithfulness CheckRAG grounding verifier~200msHallucinated claims in RAG outputs. Does the response actually derive from retrieved context?

The Output Guard uses an external classifier - never the main model self-assessing. Flagged outputs are blocked and logged. Never retried for harm violations.

Never use self-assessment: Asking the model "is your output harmful?" is unreliable by design. The model that was manipulated into producing harmful content will often evaluate its own output as clean. Use an external harm classifier. The independence is the point.

5. OWASP LLM Top 10 (2025 v2.0) Mapping

The OWASP LLM Top 10 is the industry's standard taxonomy of LLM security risks. Seven of the ten are explicitly guardrail problems - risks that cannot be mitigated by model selection, fine-tuning, or better prompts.

OWASP RiskGuard LayerMechanism
LLM01 Prompt InjectionInput GuardInjection classifier runs before the model - code that cannot itself be injected.
LLM02 Sensitive Info DisclosureInput + OutputPII detection on input (scrub/block) + PII leakage detection on output.
LLM05 Improper Output HandlingOutput GuardSchema validation and format enforcement prevent downstream misprocessing.
LLM06 Excessive AgencyPrompt GuardTool permission matrix enforced at orchestrator level per user role.
LLM07 System Prompt LeakageOutput GuardDetects system prompt content in responses before they reach the user.
LLM09 MisinformationOutput GuardFaithfulness check - synchronous verification response is grounded in retrieved context.
LLM10 Unbounded ConsumptionPrompt GuardContext budget enforcement + rate limiting in gateway. Model not involved.
The pattern: Every one of these risks has a structural solution. None of them is solved by making the prompt more emphatic about what the model shouldn't do.

7. Why This Is an Architecture Problem

The reason most LLM safety approaches fail in production is a fundamental misunderstanding of what a system prompt actually is. It's a text instruction. It lives in the context window alongside everything else the model processes - including whatever an attacker puts in the user input. The outcome depends on which instruction the model finds more persuasive in that moment.

Structural guardrails don't have that problem. Code that runs before the model doesn't care what the model thinks. A classifier that blocks prompt injections runs regardless of how sophisticated the injection is. A permission gate that prevents tool calls operates at the infrastructure level - outside the model's awareness entirely. An external harm classifier evaluates output independently of the process that generated it.

The three layers - input guard, prompt guard, output guard - are what separate a system that's mostly safe from one that's reliably safe at scale.
