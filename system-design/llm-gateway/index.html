The LLM Gateway

How to govern AI access across engineering teams - the 8-layer system that puts cost, reliability, and compliance under control

The question sounds simple: "How would you manage LLM access across 15 product teams?" Most candidates describe a spreadsheet. Or they say they'd share one API key and document the usage policy. That's a policy. It's not an architecture.

Here's what actually happens when there's no governance layer: one team's batch job wipes everyone else's monthly token budget. An OpenAI outage takes down all 15 products simultaneously because there's no fallback routing. A new compliance requirement arrives and nobody can prove what was sent to the model because there's no audit trail. And at the end of the month, the finance team gets a $47K invoice with no line-item breakdown by team or product.

This has happened at real companies. It happens every time a platform scales past a handful of services without a gateway. The LLM Gateway is how you prevent it.

The Core Architecture
Request → Auth → Cache → Route → Limit → Guard → Attribute → Retry → Audit → Response

1. Auth & Tenant: Identity First, Everything Else Second

The first thing a gateway request touches is authentication. Not routing, not rate limiting, not logging - authentication. Every request is mapped to a caller: which team, which product, which user. This mapping is the foundation of everything else in the gateway. Without it, you have no way to enforce budgets, no way to assign costs, and no way to answer an auditor's questions.

Authentication in a gateway context is more than "is this a valid API key." It's "which tenant does this key belong to, what is their current budget allocation, and what models are they permitted to call?" A read-only integration service and a real-time user-facing product have different needs, different cost profiles, and different risk tolerances. The auth layer creates that distinction before any other decision is made.

Tenant isolation is what prevents Team A's traffic from affecting Team B. Token budgets are allocated per tenant - if a team's batch job starts burning 500K tokens per hour, it hits its own limit and returns a 429. Team B's real-time product is untouched. Without tenant isolation enforced at the auth layer, one misbehaving service is a platform-wide incident.

Every request is authenticated before routing. The auth layer maps callers to tenants with budgets and model permissions - the foundation of everything downstream.

The production rule: Authenticate before routing. Route before rate limiting. Every layer downstream depends on knowing which tenant made the request. Get the identity wrong and every other control in the gateway is meaningless.

2. The Router: 10ms of Overhead, 40% Cost Reduction

Once a request is authenticated and the tenant is identified, the router decides which model gets it. This sounds trivial - but it's where most of the cost optimization happens. The default behavior without a router is that every request goes to the most capable (and most expensive) model. A customer asking "what are your hours?" goes to GPT-4o. A developer writing a complex multi-step reasoning task also goes to GPT-4o. These are not the same problem.

A complexity classifier runs on the first 200 tokens of the request in approximately 8 milliseconds. Low-complexity requests - simple Q&A, formatting tasks, short-context operations - route to GPT-4o Mini or an equivalent smaller model: 20x cheaper per token. Long-context reasoning, code generation, multi-step analysis route to the full model. The routing decision is transparent to the calling service. No application code changes when the routing logic changes.

The second function of the router is circuit breaking and fallback. When an LLM provider degrades - slow response times, elevated error rates - the circuit breaker opens and traffic shifts to a configured fallback provider automatically. No pager alert, no manual intervention, no 15-minute outage while an engineer updates environment variables. The fallback is pre-configured in the gateway, and it fires in the same request cycle.

The router adds ~10ms. Low-complexity requests go to a 20× cheaper model. If a provider degrades, the circuit breaker fires and fallback routes activate - no application code changes.

The production rule: The router adds 10ms of latency and reduces AI costs by 40% for most orgs. That math works above roughly 10,000 requests per day. Below that, a shared key is fine. Above it, the savings compound fast.

3. The Rate Limiter: Denial-of-Wallet Is a Real Attack

Request-count limits are not enough for LLM systems. They're a useful blunt instrument, but a single document analysis request consuming 50,000 input tokens costs as much in compute as 500 short FAQ queries. If your rate limiter only counts requests, a user sending 10 long requests per minute can still drain your token budget in hours.

The correct unit for LLM rate limiting is tokens per minute, split into input and output. Output tokens cost 4-5× more per token than input tokens on most providers - failing to track them separately means your rate limit is structurally underestimating real cost. A user generating 10,000-token responses at 5 requests per minute is far more expensive than the request count suggests.

The two-limit model is what prevents Friday incidents. A soft limit at 80% of the token budget sends an alert to the team's Slack channel: "you're trending toward your monthly ceiling." Engineers can investigate and adjust behavior before the ceiling hits. A hard limit at 100% returns HTTP 429 with a Retry-After header - the service degrades gracefully, not silently. Dropping requests silently is never the right behavior. The caller needs to know they've been rate-limited so they can queue, backoff, or escalate.

Soft limit at 80% triggers a Slack alert. Hard limit at 100% returns HTTP 429 with a Retry-After header. Never drop silently - silent failures hide in dead-letter queues.

The denial-of-wallet attack: A buggy retry loop running uncapped for four hours on a Friday afternoon is a $3,000 incident. It doesn't look like an attack - it looks like normal traffic, just a lot of it. A token-based hard limit stops it at $50 and sends the 429.

4. Cost Attribution: Chargeback Changes Behavior

When engineering teams have no visibility into their AI spend, they optimize for capability, not cost. They reach for GPT-4o first because it's there. They use 8,000-token prompts when 2,000-token prompts would work. They don't batch requests because the latency difference is invisible and so is the cost difference. The AI bill is someone else's problem until it shows up as a line item on the quarterly review.

When teams see their actual AI cost in real time - per request, per product, per day - behavior changes. Teams with real-time cost dashboards start reaching for smaller models first. They trim context windows. They batch requests that don't need low latency. The same engineering team makes different decisions when the feedback loop is minutes instead of months. The average cost reduction when teams move from opaque shared spending to per-team cost dashboards: 40%.

Cost attribution requires tagging every request at the auth layer with at minimum three dimensions: team, product, and user ID. These tags flow through the rate limiter, get logged with every token count, and get rolled up into daily and monthly chargeback reports. The weekly report delivered to each team's engineering lead - "here's what you spent last week and how it compares to budget" - is the behavioral mechanism. It's not accounting. It's a feedback loop.

Tag every request at auth. Log token counts with attribution at every LLM call. Feed real-time dashboards and weekly chargeback reports. The visibility is the mechanism.

The production rule: Cost attribution is not an accounting feature. It's an engineering behavior change mechanism. Teams with opaque spend optimize for capability. Teams with real-time spend visibility optimize for cost. Same engineers, completely different decisions.

5. Audit & Observability: Log What Matters, Scrub What Doesn't

In regulated industries, an AI audit isn't optional - it's contractual. When a healthcare system routes patient documents through an LLM and an adverse outcome occurs, someone will ask: what was sent to the model? What did it respond? Who authorized the call? Which model version was used? Without a gateway, the answer to all of these is "we don't know." That is a compliance failure before the conversation even begins.

The audit layer captures structured request metadata for every gateway call: tenant, team, user ID, model requested, model actually used, input token count, output token count, latency, response status, timestamp. What it does not log: the plaintext prompt. PII in your audit log is PII leakage with extra steps. Log hashes of inputs, not raw content. Retain full encrypted inputs separately with a short TTL - 48 hours is enough for debugging; 90 days is the minimum for most compliance frameworks.

Observability means three key metrics tracked continuously: latency P50 and P95 per model, because provider degradation shows up in latency before it shows up in errors. Error rate per tenant, distinguishing client-side validation errors from provider-side failures. And budget burn rate - a simple projection: "at the current spend rate, Team A hits their ceiling on day 19 of the month." Catch it on day 5 and there's time to act. Catch it on day 18 and it's a crisis.

PII is scrubbed before the audit write. The log captures metadata - not plaintext. Latency and burn rate monitoring catch provider issues and budget overruns before they become incidents.

The production rule: Log hashes, not plaintext. Retain full encrypted inputs separately with a 48-hour TTL for debugging. The 90-day audit log is for compliance - it needs team, model, tokens, and timestamp, not the raw prompt content.

7. Build It Before the Invoice Arrives

The LLM Gateway is infrastructure that feels optional until it suddenly isn't. Teams skip it when they're small - one service, one team, one API key, no problem. They skip it when they're moving fast - there are features to ship and the governance layer can wait. They skip it until a $47K invoice arrives with no breakdown, or an OpenAI outage takes down three product teams simultaneously, or the compliance team asks to see what was sent to the model last quarter and the answer is "we don't know."

The eight layers - auth and tenant isolation, semantic caching, intelligent routing, token-based rate limiting, guardrails and safety, cost attribution, retry and fallback, and PII-scrubbed audit logging - address eight distinct categories of production risk. None of them are engineering elegance for its own sake. Each one exists because a specific thing went wrong at a real company without it.

Build the gateway before you need it. Retrofit it after 15 teams are calling LLMs in production and you'll spend twice as long while also being half as thorough. The time to build governance infrastructure is before the complexity arrives. Not after.
