Context Engineering: The 5-Layer Stack Behind Every LLM Call

Most engineers optimize the prompt. The ones building production agents manage the entire context window.

The answer your model gives is decided before it generates a single token. By the time inference starts, the outcome is already shaped by what sits in the context window: your instructions, your examples, the retrieved chunks, the conversation history, and the agent's own scratchpad. The model does not reason in a vacuum. It reasons over whatever you assembled and handed it.

This is why prompt engineering hits a ceiling. Your system prompt might be a few hundred well-crafted tokens. But it is sharing the window with retrieval results, prior turns, and tool outputs that add up to tens of thousands of tokens. Tuning the 5% you wrote while ignoring the other 95% is optimizing the label on a bottle while ignoring what is inside it.

Context engineering is the discipline of managing all of it. Every LLM call is a context assembly problem: which layers get to contribute, how many tokens each is allowed, and what gets trimmed when the budget runs out. Get this right and a mediocre prompt still ships. Get it wrong and a perfect prompt still hallucinates.

The Core Principle
Context = Instructions + Examples + Retrieved + History + State

1. What Actually Fills the Context Window

Open up a single call to a production agent and look at the token breakdown. It rarely matches anyone's intuition. The part you wrote and tested - the system prompt - is almost always the smallest slice of the input. Everything else is dynamic, assembled at request time, and changing on every call.

A representative agent turn looks like this: a system prompt of around 500 tokens, the part you authored and the only stable piece. A set of retrieved chunks at around 3,000 tokens - RAG results, search hits, database lookups - different on every call, and treated by the model as ground truth. And a conversation history plus tool output of 8,000 tokens or more, growing with every interaction, that nobody is trimming. The 500 tokens you obsess over are roughly 4% of what the model actually reads.

The system prompt is the smallest and most stable slice. The dynamic layers - retrieval and history - dominate the window and decide the output.

Key insight: Prompt engineering tunes the wrapper. Context engineering tunes the full input. If you can only improve one, improve the layer that consumes the most tokens, not the one you find easiest to edit.

2. The 5-Layer Context Stack

Every LLM call is assembled from five distinct layers. Two are static - written once and rarely touched. Three are dynamic - rebuilt on every single call. Naming them is the first step to managing them, because you cannot budget what you have not separated.

#LayerWhat lands hereTypical budget
01System InstructionsTask, role, constraints, output format - what you write directly5-15%
02Few-Shot ExamplesDemonstrations of the target output format and behavior5-10%
03Retrieved KnowledgeRAG chunks, search results, tool outputs, DB lookups20-60%
04Conversation HistoryPrior turns - the biggest budget consumer over time30-80%
05State and ScratchpadAgent reasoning, tool call chains, intermediate outputsvariable

Five layers fan into one window. Layer 3 carries the highest variance and the highest leverage. The model only ever sees the assembled result.

3. Layers 1 and 2: The Control Plane

System instructions and few-shot examples are the static layers. You write them once, they rarely change between calls, and together they usually consume well under 1,000 tokens. They are small. They are also the easiest layer to get quietly wrong, because their mistakes do not stay contained - they cascade.

System instructions define the task, the role, the constraints, and the output format. This is the part everyone calls "the prompt." It is high-leverage precisely because it is small: a single ambiguous constraint here does not just produce one bad answer, it breaks how every downstream layer is interpreted. A vague instruction about output format makes retrieval harder to use and history harder to trim, because nothing downstream knows what "done" looks like.

Few-shot examples are demonstrations of the behavior you want. Three concise examples of your target output can anchor structure in under 300 tokens, where two paragraphs of prose describing the same format consistently underperform. Models imitate examples more reliably than they follow descriptions. When output structure matters, show, do not tell.

Key insight: The static layers anchor behavior. The dynamic layers - 3 through 5 - determine quality per call. Spend real effort on layers 1 and 2 once, then stop touching them and move your attention to where the tokens actually live.

4. Layer 3: Retrieved Knowledge Is Where Quality Is Won or Lost

Layer 3 is the most variable layer in the entire stack. RAG chunks, search results, tool outputs, database lookups - it is different on every single call, and the model treats whatever lands here as ground truth. This is the single highest-leverage surface in context engineering, and it is where most debugging effort belongs.

The asymmetry is the whole point. The right retrieved content can compensate for a weak prompt - if the model is handed the exact passage it needs, even a mediocre instruction produces a good answer. The reverse is never true. No amount of prompt polishing fixes retrieval that surfaced the wrong document. A perfect system prompt sitting next to last quarter's pricing sheet will produce a confident, well-formatted, completely wrong answer about the price.

That confident-wrong failure mode is what makes Layer 3 dangerous. The model does not hedge when retrieval is bad. It reads the stale document, treats it as fact, and defends the wrong answer with full fluency. Which is why for Layer 3, retrieval precision beats retrieval recall: one exactly-right chunk is worth more than ten loosely-related ones, because the loosely-related ones dilute attention and invite the model to anchor on the wrong detail.

Warning: When a model with a good system prompt still produces bad answers, debug retrieval before you touch the prompt. A stale or wrong chunk in Layer 3 overrides good instructions every time. The model follows what it reads, not what you told it.

5. Layers 4 and 5: History and State, Where Agents Break Over Time

Here is the failure every agent builder eventually meets: the agent works perfectly at turn 5 and falls apart at turn 20. Nothing about the prompt changed. Nothing about the model changed. What changed is the context window filled up.

By turn 20, the conversation history that started at 1,000 tokens has grown to 15,000. Prior turns, tool call results, and the agent's own backtracking now consume most of the window. Your carefully written 500-token system prompt is no longer near the top of the input - it is sandwiched in the middle, between thousands of tokens of old conversation. And models attend less reliably to the middle of a long context than to its edges, the well-documented "lost in the middle" effect. The instructions are technically still there. The model has effectively stopped reading them.

The fix is not a better prompt. The fix is a trimming strategy. A sliding window keeps the most recent N turns verbatim. Older turns get summarized into a compact running summary rather than carried in full. Each layer gets an explicit token budget, and when the budget is exceeded, something is dropped or compressed on purpose rather than by accident. Context window management is memory management with a hard cap - and like all memory management, doing nothing is itself a decision, just a bad one.

StateHistory sizeWhat happens to the system prompt
Turn 5~1K tokensNear the top of the window, fully attended
Turn 20 (no trimming)~15K tokensBuried in the middle, effectively ignored
Turn 20 (with trimming)~4K tokensStill near the top, recent turns summarized
Key insight: An agent that degrades as the conversation gets longer almost never has a prompt problem. It has a trimming problem. Add a sliding window and turn summarization before you rewrite a single instruction.

6. The Three Failure Modes

Context engineering fails in production in three recognizable ways. Naming them turns vague "the agent got worse" reports into a specific layer to go fix.

Failure modeWhat goes wrongThe fix
Context stuffingRaw documents dumped into the window. Attention dilutes across noise and the model acts on the wrong parts. A support agent retrieves 10 FAQ pages instead of the 1 relevant answer.Chunk compression and MMR reranking. Maximize signal density, not token count.
History bloatNo trimming strategy. A coding agent at turn 30 carries 20K tokens of old tool output nobody needs, and the system prompt lands in the forgotten middle.Sliding window plus turn summarization with a per-layer token budget.
Layer interferenceA single 40K-token tool result pushes the system instructions out of prime attention. The model follows the tool output instead of your rules.Priority ordering by recency times relevance, and a hard cap on any single layer.
Warning: More context only helps while signal density stays high. Past that point, every extra token of low-signal context actively dilutes attention and makes the output worse. Bigger context windows did not end this problem - they raised the ceiling on how badly you can stuff one.

7. The System Design Consequence

Once you see the stack, the standard agent components stop looking like separate features and start looking like what they actually are: machinery for deciding what goes in each layer. Every component in an agent architecture is, underneath, a context router.

Memory decides which prior turns stay in the window and which get summarized or dropped. It is not "store more history" - it is selective history, a Layer 4 budget manager. Retrieval is a Layer 3 budget optimizer whose real job is maximizing signal density per token, not dumping every match it finds into the window. And the orchestrator is the thing that assembles the final context for each LLM call - its core question, every single turn, is "what goes in each layer this turn?"

This reframes the model itself. The LLM is a function. Context engineering is how you write the input to that function. Memory stores context, retrieval selects context, orchestration assembles context - and the quality of the output is mostly decided by that assembly, not by the function.

The Input Is the System

The model is the part everyone optimizes because it is the part everyone can see. But the model is fixed - you call the same API everyone else calls. What you actually control is the input: which of the five layers contributes, how many tokens each gets, and what is trimmed when the window fills. That assembly is where production agents are won or lost.

The next time an LLM system misbehaves, resist the reflex to rewrite the prompt. Ask which layer failed instead. Was the retrieval wrong? Did history bury the instructions? Did one giant tool output drown out everything else? Name the layer, fix that layer, and you will spend far less time tuning prompts that were never the problem.

The model is a function. Context engineering is how you write its input. Manage all five layers.
