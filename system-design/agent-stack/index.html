The 5-Component Agent Stack: The 80% Engineers Miss

Everyone builds the model call. Almost nobody designs the system around it.

Everyone builds the model call. Almost nobody designs the system around it - the orchestrator, tools, memory, policies, and observability that determine whether it actually ships.

Walk into any system design interview and ask someone to design an AI coding assistant. The answer is almost always the same: "I'd use GPT-4 with a few tool-use prompts." That's an API call. It's not a system design.

The LLM is stateless. It has no memory of last week's conversation, no ability to call your database, no awareness of what actions it's already taken this session, and no enforcement of what it's allowed to do. Those things don't live in the prompt. They live in the five components that make a language model into an agent - the system around it.

This is what separates a demo that runs on localhost from an agent that ships to production. The model is 20% of the problem. Here's the other 80%.

The Core Principle
Agent = Orchestrator + Tools + Memory + Policy + Observability

1. The Orchestrator: Who Decides When the Loop Stops?

The orchestrator is the control plane of your agent. It receives the input, plans which action comes next, invokes the LLM or a tool, processes the result, and makes exactly one decision: loop back for another step, or return the answer. Nothing in the LLM controls this loop. The LLM reasons. The orchestrator decides.

This separation matters because the loop is where agents go wrong in production. Without a hard iteration cap, an agent chewing on a complex problem can run 40 steps, spend $8 in inference costs, and still not finish. The orchestrator owns that cap. It owns tool call sequencing - ensuring a retrieval step runs before a synthesis step, not after. It owns error recovery: if a tool times out, does the agent retry, skip it, or fail gracefully? All of that is orchestrator logic. None of it belongs in the model.

The most important thing the orchestrator does is separate business logic from reasoning. Business rules - "always log user actions," "never call the billing API more than once per session," "escalate to a human if confidence is below 0.7" - live in the orchestrator, in code, where they're testable and deterministic. The model thinks. The orchestrator governs.

The orchestrator owns the loop. The LLM reasons inside a single step. Iteration cap, error recovery, and tool sequencing all live in the orchestrator - not the prompt.

The production rule: Set a hard iteration cap of 10-20 steps in the orchestrator. An agent that loops without a bound isn't reasoning deeper - it's burning cost and accumulating risk. Cap it in code, not in the prompt.

2. The Tool Layer: The LLM Can Talk, It Can't Act

The LLM generates text. That's it. Every real-world action the agent takes - searching the web, querying a database, running code, calling an API, writing a file - happens through the tool layer. Tools are how the model connects to the world outside its context window. Without them, an agent is just a chatbot with long system prompts.

Every tool needs four things, and skipping any of them creates a production incident. First, a schema: the exact parameters the tool accepts, typed and validated. The LLM generates tool calls as JSON - if the schema is loose, the calls will be malformed. Second, timeout and retry logic: tools fail. Networks drop. Databases are slow. The agent must handle failures explicitly, not inherit whatever error behavior the underlying API provides. Third, a permission gate: can this user, in this role, call this tool? Permissions must be explicit. If the check is missing, eventually the agent will try something it shouldn't. Fourth, a defined result format: structured JSON back to the model, not raw text. Structured output is easier to parse, easier to validate, and harder for the next reasoning step to misinterpret.

The permission gate is where most teams cut corners. They build the tools, they work in testing, and they add permissions "later." Later doesn't arrive until a user does something unexpected with a capability they should never have reached.

Every tool call passes schema validation, a permission check, and timeout handling before it executes. Structured JSON returns to the LLM - not raw unstructured text.

The production rule: If the tool can do something and no permission gate exists, the agent will eventually attempt it. Build the permission matrix before you build the tools. Not after.

3. Memory: The Context Window Is a Workspace, Not Storage

Every turn, your agent starts cold. The LLM has no memory of the previous session, no knowledge of what the user told you last week, and no awareness of anything outside the current context window. You have to build memory. And memory in an agent system is not one engineering problem - it's three distinct ones.

Working memory is everything in the current context window right now: the system prompt, the conversation history from this session, the tool results from recent steps. It's immediate - no retrieval needed, it's already there. But it's bounded. A 128k context window sounds massive until you're running a multi-step agent with verbose tool outputs and 40 turns of history. Working memory overflows. You need an explicit eviction strategy.

Short-term memory is the session-scoped conversation store: the last N turns of this user's interaction, persisted to a database, retrieved at the start of each request and injected into the context. This is what lets a user say "like I mentioned earlier" and have the agent know what they mentioned. Without it, every new request resets the entire conversation to zero - the agent is functionally amnesiac between calls.

Long-term memory is semantic search over a vector store: past conversations, user preferences, domain knowledge, historical decisions. Retrieved by similarity to the current query, not by recency. This is the expensive component - both to build and to evaluate - but it's what makes an agent feel persistent and intelligent over weeks and months of use.

Memory is three separate engineering problems. What's in the window right now. What's in the session DB. What's retrievable via semantic search across all past interactions.

The context window is not memory. Storing full conversation history in the context window works in a demo. Under long sessions it overflows. Build a conversation store and a retrieval layer from day one - not when the demo breaks at 50 turns.

4. The Policy Engine: Prompts Are Suggestions, Architecture Is Law

The most common security mistake in agent systems is putting safety constraints in the system prompt. "Don't do harmful things." "Never access files outside the /data directory." "Only call approved tools." These are suggestions. A clever prompt injection can override them. A model behavior update can shift how they're interpreted. They are not guarantees, and you cannot rely on them for enforcement in a production system.

The policy engine is structural enforcement. It runs before the prompt is assembled, before the LLM is called, before any tool executes. It answers exactly one question: is this action allowed for this user in this context? If no, the action doesn't happen, regardless of what the model generated or what was in the prompt.

Concrete structural policies look like: a read-only user role cannot call write-capable tools. A specific tool is capped at three calls per session. Any action that deletes records requires explicit human confirmation before execution. No user data crosses region boundaries, regardless of what the request contains. None of these belong in a system prompt. All of them belong in code, evaluated before any LLM inference happens.

Blocked actions return 403 before any model call happens. No prompt engineering bypasses the policy engine - it's code, not a language model.

The production rule: If a constraint needs to be guaranteed rather than merely suggested, it belongs in the policy engine, not the system prompt. Structural enforcement is code. Prompts can be jailbroken. Code cannot.

5. Observability: You Can't Debug What You Can't Trace

Traditional application monitoring fails completely for agents. An HTTP request returns 200. Your logs show success. But inside that request, the agent ran 8 iterations, one tool returned empty results on iteration 3, the model was called 6 times at 40 cents per call, and the final answer was derived from a hallucinated tool result on iteration 7. None of that is visible in standard request logs.

Agent observability requires step-level tracing through the entire decision loop - not just at the HTTP boundary. That means capturing every LLM call with its full prompt, completion, latency, model used, and token count broken down as input and output. Every tool call with its input parameters, the output it returned, duration, and error state. Every orchestrator decision - which action was chosen, what iteration number, what the agent's internal reasoning state was. And cost per session, not just per request - because a single session can have wildly variable cost depending on how the agent reasoned its way to an answer.

Step-level traces are what let you answer the two most common agent debugging questions: "why did it loop 10 times?" and "why did this session cost $3?" Without the traces, you are guessing. With them, the answer is usually visible in 30 seconds.

Step-level traces expose what's invisible in HTTP logs: which tool was slow, which LLM call was expensive, how many iterations the agent needed to converge.

The production rule: Track cost per session - not per request. Roll up token counts from every LLM call within the session. A single runaway session that loops 40 times will not appear unusual in per-request metrics. It will be obvious in per-session cost data.

The System Is the Hard Part

The LLM is the most legible part of an agent - it's what you see in the demo, what you show stakeholders, what gets mentioned in every engineering blog post. But it's not where production agents fail. They fail because the orchestrator has no iteration cap and loops 40 times. They fail because a tool returned an error and no recovery path exists. They fail because a user's session data wasn't retrieved and the agent had no context for what was asked. They fail because a permission wasn't checked and someone did something they shouldn't have been able to do. They fail silently at 3am in ways that only become visible in the next morning's cost report.

The five components - orchestrator, tool layer, memory, policy engine, observability - are not architectural luxuries. They are the production architecture. An agent without them is a language model with a long system prompt and no safety net. Every fundamental limitation the model has - no memory, no real-world actions, no enforcement, no traceability - is solved by a specific component, not by a better prompt.

The model is the easy part. Design the other 80% first.
